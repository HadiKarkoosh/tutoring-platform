# منصة الدروس الخصوصية

منصة حجز دروس: طلاب يتصفحون المدرّسين حسب المادة أو الاسم ويحجزون مواعيد متاحة ويقيّمون تجربتهم، ومدرّسون يديرون ملفهم الشخصي ومواعيدهم وحجوزاتهم.

## التقنيات

| الجزء | التقنية |
|---|---|
| Backend | Nest.js + TypeORM + SQLite (better-sqlite3) + JWT + Jest |
| Frontend | Next.js 14 (App Router) + React 18 |

## التشغيل

### 1. الباك اند (المنفذ 4000)

```bash
cd backend
npm install
cp .env.example .env   # اختياري بالتطوير — لازم بالإنتاج (شوف "متغيرات البيئة")
npm run start:dev
```

قاعدة البيانات SQLite تُنشأ تلقائياً (`tutoring.sqlite`) مع بذر المواد الافتراضية، وأول تشغيل على قاعدة فاضية بيبذر كمان 4 مدرّسين تجريبيين بأسماء وموادّ مختلفة (نفس القيمة التعليمية موجودة فوراً بالصفحة الرئيسية بدون ما تسجّل حساب).

## متغيرات البيئة

- Backend (`backend/.env.example`): `PORT` (افتراضي 4000)، `JWT_SECRET` (**إجباري بالإنتاج** — بدون قيمة الباك اند بيرفض يشتغل)، `DB_PATH`، `FRONTEND_URL` (لبناء رابط إعادة تعيين كلمة المرور)
- Frontend (`frontend/.env.example`): `NEXT_PUBLIC_API_URL` (افتراضي http://localhost:4000)

## إعادة تعيين كلمة المرور

بيئة تجريبية بدون خدمة بريد إلكتروني فعلية — بدل ما `POST /auth/forgot-password` يبعت إيميل، بيرجّع رابط إعادة التعيين مباشرة بالـ response (`resetUrl`)، والفرونت اند بيعرضه بصفحة "نسيت كلمة المرور" مع توضيح إنو هاد سلوك بيئة تجريبية. بنشر حقيقي، هالرابط لازم يترسل بالإيميل فقط ويُحذف من الـ response.

## منع التعارض (Double-booking)

الحجز يتم داخل Transaction واحدة:

1. **تحديث شرطي ذرّي**: `UPDATE slot SET isBooked = 1 WHERE id = ? AND isBooked = 0` — من طلبين متزامنين، واحد فقط يُصيب صفاً والثاني يستلم **409 Conflict**.
2. **قيد فريد** على `Booking.slotId` (unique JoinColumn) — شبكة أمان ثانية على مستوى قاعدة البيانات نفسها.
3. **قفل (Mutex) على مستوى العملية** حول عمليتي الحجز والإلغاء: `better-sqlite3` هو اتصال واحد متزامن (synchronous)، فلو صارت معاملتان (transactions) متداخلتان بنفس اللحظة بيرفض SQLite الثانية بدل ما ينتظرها. القفل بيسلسل الوصول لهالاتصال حتى يضل التحديث الذري فوق هو يلي يحسم مين ربح الحجز، بدل ما تفشل الطلبين. مغطى باختبار آلي (`bookings.service.spec.ts`) بيطلق حجزين متزامنين حقيقيين على نفس الموعد ويتأكد إنو واحد بس نجح.

## اختبار سريع للسباق (يدوي)

شغّل الباك اند ثم نفّذ طلبين متزامنين على نفس الموعد بحسابي طالبين — النتيجة: واحد 201 والثاني 409، وحجز واحد فقط بقاعدة البيانات. نفس السيناريو مغطى آلياً بـ `npm test`.
